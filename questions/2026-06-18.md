**17:35 · 第 01 章 · 火车票查询工具设计点评**

<student>
1.工具描述：可以查询火车票 2.始发地、目的地、日期、余票 3.日期必须小于查询当日，余票必须为整数 4.成功返回始发地、目的地、日期、余票，失败返回报错信息 5.每日凌晨调用，做出差计划时模型自动选择调用
</student>

<response>
整体结构是对的：你已经覆盖了描述、输入、校验、结果和调用策略。但有三个关键修正。

### 1. 工具描述需要说明“何时用、何时不用”

“可以查询火车票”太宽泛。可以改成：

> 查询指定出发地、目的地和乘车日期的实时火车班次及余票。用户询问火车出行或制定出差计划时使用；不用于购买、退改签或查询历史班次。

这样模型不会误以为它还能买票。

### 2. `余票` 应该是输出，不是输入

用户提供的是查询条件，余票是工具查询后得到的结果。

输入字段应该是：

```json
{
  "origin": "北京南",
  "destination": "上海虹桥",
  "travel_date": "2026-06-20"
}
```

工具执行后，才返回车次和余票数量。

### 3. 日期规则方向反了

你写的是“日期必须小于查询当日”，这会变成查询过去的车票。应该是：

- 乘车日期不得早于今天；
- 乘车日期不得超出售票系统允许的预售范围；
- 始发地不能等于目的地；
- 两个车站必须真实存在。

“余票必须为非负整数”是对**工具返回数据**的校验，不是对用户输入的校验。

### 4. 成功和失败结果应便于程序判断

成功示例：

```json
{
  "ok": true,
  "origin": "北京南",
  "destination": "上海虹桥",
  "travel_date": "2026-06-20",
  "trains": [
    {"train_no": "G1", "available_seats": 12}
  ]
}
```

失败示例：

```json
{
  "ok": false,
  "error_code": "SERVICE_TIMEOUT",
  "retryable": true,
  "message": "余票服务暂时无响应"
}
```

注意：查询成功但没有余票，仍然是成功结果，`trains` 可以为空；这不属于系统错误。

### 5. 定时调用和模型选择是两个不同概念

“每日凌晨调用”属于第 20 章的**主动触发或定时调度**，不是 `tool_choice`。如果每天必须执行固定查询，调度器甚至可以直接调用工具，不必让模型决定。

在对话中：

- 用户只是闲聊时，不调用；
- 用户说“帮我规划北京到上海的出差行程”时，可以让模型自动选择；
- 用户点击“查询余票”按钮时，应用程序可以强制调用这个工具。

面试版总结：你的设计方向正确，但需要严格区分三样东西——**用户提供的输入、工具查询出的结果、触发工具运行的机制。**这正是第 01 章最重要的边界。
</response>

**17:31 · 第 01/03 章 · 语义校验与结构校验**

<student>
什么事语义校验规则
</student>

<response>
**语义校验规则**检查的不是“数据格式对不对”，而是“这个值在真实业务中是否合理、存在、允许”。

例如 `create_todo` 收到：

```json
{
  "title": "复习第01章",
  "due_at": "2020-01-01T09:00:00"
}
```

它通过了结构校验：

- `title` 是字符串；
- `due_at` 是合法日期格式；
- 必填字段都存在。

但它可能无法通过语义校验，因为截止时间已经过去。

两者可以这样区分：

| 校验类型 | 检查的问题 | 例子 |
|---|---|---|
| Schema／结构校验 | 数据“长得对不对” | 字段是否存在、类型是否正确、格式是否合法 |
| 语义校验 | 数据“在现实中讲不讲得通” | 时间是否在未来、用户是否有权限、ID 是否真实存在 |

`create_todo` 可以有这些语义规则：

- `title` 去掉空格后不能是空字符串；
- 提醒时间必须晚于当前时间；
- 指定的项目 ID 必须真实存在；
- 当前用户必须有权向该项目添加待办；
- 相同请求不能重复创建待办。

再看一个极端例子：

```json
{"hour": 99}
```

如果 schema 只规定 `hour` 是整数，它会通过结构校验；但一天没有第 99 小时，所以语义上无效。你可以在 schema 中增加 `0–23` 的范围约束，但“这个时间是否已经过去”仍然必须结合当前时间在代码中判断。

一句话记忆：**Schema 判断输入是否合法，语义校验判断请求是否合理且可执行。**

回到第 5 题，你可以先给 `create_todo` 写两条最简单的语义规则：标题不能为空，提醒时间必须在未来。
</response>

**17:28 · 第 01 章 · 超时错误与有限重试**

<student>
返回超时异常，设定最大重试次数
</student>

<response>
核心正确。更严谨地说：**捕获超时异常，把它转换成结构化的错误 `tool_result`，而不是把异常继续抛出。**

例如：

```json
{
  "tool_use_id": "call_123",
  "is_error": true,
  "error_code": "TIMEOUT",
  "retryable": true,
  "content": "天气服务在 5 秒内没有响应"
}
```

然后由循环控制：

- 最多重试 2 次；
- 每次重试前短暂退避；
- 超过次数后停止，并向用户说明失败原因。

还有一个生产级加分点：不是所有工具都能安全重试。`get_weather` 是读取操作，通常可以重试；`create_todo`、付款等写入操作可能造成重复记录，需要幂等键或先查询执行状态。

一句话面试答案：

> 工具层应捕获超时异常并返回带调用 ID、错误类型和可重试标志的工具结果；循环通过最大重试次数和时间预算防止无限重试，写操作还要保证幂等性。

**17:26 · 第 01 章 · 防止 Schema 漂移**

<student>
会执行错误，参数错误。可以schema做弱校验
</student>

<response>
“会发生参数错误”是对的。例如分发代码执行：

```python
get_weather(city="Tokyo")
```

但函数签名已经变成：

```python
def get_weather(location: str):
    ...
```

Python 通常会抛出类似 `unexpected keyword argument 'city'` 的 `TypeError`。

不过，schema 校验不能解决这次问题。因为模型返回的 `city` **完全符合旧 schema**，校验会顺利通过；真正过期的是 schema 与函数之间的契约。

防止 schema 漂移可以分三层：

1. **把 schema 和 handler 绑定成一个工具定义**，修改时一起改；
2. **使用单一数据模型生成 schema**，例如用 Python 的 Pydantic 模型生成 JSON Schema，handler 也接收同一个模型；
3. **编写契约测试**，用符合 schema 的示例参数实际调用 handler，确保不会出现参数名不一致。

例如：

```python
class WeatherInput(BaseModel):
    location: str

def get_weather(args: WeatherInput):
    ...
```

工具 schema 从 `WeatherInput` 自动生成，处理函数也使用 `WeatherInput`，这样只有一个参数定义来源。

注意：即使开启服务商的严格 schema 模式，也只能保证模型遵守那个 schema，不能保证 schema 与 Python 函数同步。

一句话记忆：**校验防止坏参数，单一数据源和契约测试防止坏契约。**

</response>

**17:23 · 第 01 章 · 工具调用 ID 的关联作用**

<student>
1.app将用户的promt“查询实时天气”和可以调用的function list发送给llm 2.llm判断是否可以使用查询天气的function，并返回调用工具的请求和function schema 3.app执行function，返回weather result 4.app将weather result发送给llm。不知道id为什么必须原样返回
</student>

<response>
四步主线正确。第 2 步需要修正：模型不会把 function schema 再返回，因为 schema 已经由 App 在第 1 步发送给模型了。模型返回的是：

```json
{
  "id": "call_123",
  "name": "get_weather",
  "arguments": {"city": "Tokyo"}
}
```

更准确的四步是：

1. App 发送用户消息和可用工具的 schema；
2. LLM 判断是否需要工具，并返回调用 ID、工具名称和参数；
3. App 校验参数、执行函数并取得结果；
4. App 使用同一个调用 ID 把结果送回 LLM，LLM 再生成最终答案。

`id` 可以理解为**订单号**。

假设模型同时请求：

```text
call_123 → 查询东京天气
call_456 → 查询北京天气
```

工具结果可能以不同顺序返回。App 必须告诉模型：

```text
call_456 → 25°C
call_123 → 18°C
```

模型才能知道哪个结果属于哪个请求。如果丢失、写错或交换 ID，模型就无法可靠匹配请求和结果；一些 API 会直接拒绝后续消息，并报告存在未回答的工具请求。

所以必须原样返回 ID，因为它保证了：

- 请求和结果能够一一对应；
- 并行调用不会串结果；
- 每一个工具请求都确实得到了回答。

一句话记忆：**工具名称决定“做什么”，调用 ID 决定“这是谁的结果”。**

现在进入第 3 题：schema 中的参数叫 `city`，Python 函数后来改成接收 `location`，但 schema 没改。运行时会发生什么？你会如何防止这种问题？
</response>

**17:15 · 第 01 章 · 单次工具调用的真实场景**

<student>
关于01 chapter。Give me three real-world examples of where this matters.
</student>

<response>
第 01 章真正重要的地方是：**让模型从“只会说”变成“能够通过可靠代码获取信息或执行动作”。**

### 1. 个人助理创建待办

用户说：“提醒我明早 9 点复习第 01 章。”

- 模型理解意图，生成 `create_todo(title, time)` 请求；
- Python 代码校验时间并写入数据库；
- 工具返回待办编号；
- 模型回答：“已创建明早 9 点的提醒。”

为什么需要工具：模型自己说“已创建”并不代表提醒真的存在。只有代码成功写入后，它才能诚实确认。这最贴近你想做的 Hermes 类助理。

### 2. 电商客服查询订单

用户问：“我的订单为什么还没到？”

- 模型请求调用 `get_order_status(order_id)`；
- 后端查询真实物流系统；
- 工具返回“运输中，预计明天到达”；
- 模型把结果解释给用户。

为什么需要工具：订单状态是实时且私有的数据，不在模型训练知识中。如果没有工具，模型只能猜测。

### 3. 编码智能体运行测试

用户说：“检查这个项目的测试是否通过。”

- 模型请求调用 `run_tests()`；
- 程序真正执行测试命令；
- 工具返回退出码和失败信息；
- 模型根据真实结果总结问题。

为什么需要工具：模型仅阅读代码无法证明测试通过。它必须运行命令，并以执行结果为依据。

这三个例子分别代表工具调用的三种核心用途：

- **执行动作**：创建待办；
- **读取外部状态**：查询订单；
- **验证现实结果**：运行测试。

共同原则是：**模型负责理解和决策，工具负责接触真实世界。**
</response>

**17:09 · 第 01 章 · 服务商特定控制项**

<student>
解释下值得了解的服务商特定控制项
</student>

<response>
“服务商特定控制项”是指：**工具调用的四步机制相同，但 OpenAI、Anthropic、Gemini 等提供了不同的开关，用来控制模型怎样调用工具。**

| 控制项 | 控制什么 | 适合什么时候用 |
|---|---|---|
| `tool_choice` | 模型是否可以调用工具、必须调用工具，或必须调用指定工具 | 当业务流程已经确定，不想让模型自由决定时 |
| 并行工具调用 | 一次让模型请求多个工具 | 多个任务互不依赖时，例如同时查天气和日历 |
| 严格 Schema 模式 | 强制参数在结构上符合 JSON Schema | 希望减少缺字段、类型错误时 |
| 结构化输出 | 强制模型最终答案符合指定 JSON 结构 | 最终需要数据而不是执行动作时 |
| 托管工具 | 由模型服务商执行搜索、代码运行等工具 | 希望快速接入能力，不想自己实现执行器时 |
| 拒绝与内容过滤 | 模型因安全策略拒绝执行请求 | 必须作为正常分支处理，不能假设每次都会调用工具 |

### 1. `tool_choice`：谁来决定是否调用工具

常见模式可以理解为：

- `auto`：模型自行判断是否调用工具；
- `none`：禁止调用工具，只能回答文本；
- `required`：必须调用某个工具；
- 指定工具：必须调用 `create_todo` 等特定工具。

例如用户点击“创建待办”按钮时，系统已经明确知道意图，可以直接要求模型调用 `create_todo`，没必要再让它猜。

### 2. 并行工具调用：一次开多张工具单

用户问“查看明天天气和我的日程”，模型可以同时请求：

```text
get_weather(date="tomorrow")
get_calendar(date="tomorrow")
```

如果两者互不依赖，并行可以降低延迟。如果第二个工具需要第一个工具的结果，就不能并行。你的第一版助理可以先关闭并行，把循环做正确后再加入。

### 3. 严格 Schema 模式：保证结构正确

假设工具要求：

```json
{"title": "复习第01章", "hour": 9}
```

严格模式可以防止模型把 `hour` 写成字符串，或者漏掉必填字段。但它只能保证**结构正确**，不能保证**语义正确**——`hour: 99` 仍需要你的代码校验。这正是第 03 章会讲的边界验证。

### 4. 结构化输出：要数据，不要动作

它和工具调用很像，但目的不同：

- 工具调用：要求程序执行动作，例如创建待办；
- 结构化输出：要求模型整理数据，例如从一段文字提取待办列表。

一句话区分：**要做事，用工具；要 JSON，用结构化输出。**

### 5. 托管工具：服务商替你执行

网络搜索、代码执行、文件检索等能力，可以由服务商直接运行。优点是接入快；缺点是执行过程、费用和数据边界更难完全控制。涉及本地笔记、私人日历时，你通常仍然需要自己实现工具。

### 6. 拒绝与内容过滤：工具调用不保证发生

即使你要求调用工具，模型也可能因为安全策略拒绝。程序应该记录并向用户展示拒绝原因，而不是把它当成无法解析的异常或不断重试。

对你的 Hermes 类助理 V0，我建议：

- `tool_choice` 先用自动模式；
- 暂时关闭并行调用；
- 能用严格 Schema 就使用，但仍做语义校验；
- 本地待办和记忆使用自定义工具；
- 把拒绝、超时和工具错误都作为正常结果处理。

这些控制项的具体字段名会随服务商和 SDK 变化，但背后的设计问题不会变：**模型有多少选择权、参数有多可靠、工具由谁执行、失败后怎样继续。**
</response>

**17:06 · 第 01 章 · 工具调用的四步循环**

<student>
解释一下四部循环
</student>

<response>
“四步循环”的核心是：**模型负责决定做什么，Python 代码负责真正执行。**

以“东京现在天气怎样？”为例：

1. **把消息和工具说明发给模型**

   你的程序把用户问题，以及 `get_weather(city)` 的名称、用途和参数格式一起发给模型。

2. **模型返回工具请求**

   模型不直接查询天气，而是返回类似：

   ```json
   {"id":"call_123","name":"get_weather","arguments":{"city":"Tokyo"}}
   ```

   这只是请求，不是函数已经被执行。

3. **你的 Python 程序执行工具**

   程序校验参数，根据名称找到函数，然后真正运行：

   ```python
   result = get_weather(city="Tokyo")
   ```

   假设结果是 `18°C，局部多云`。

4. **把工具结果送回模型**

   程序使用相同的 `id` 把结果发回去。模型读到结果后，生成自然语言答案：“东京目前 18°C，局部多云。”

可以把它记成：

`用户提问 → 模型开工具单 → Python 执行 → 模型组织答案`

最容易混淆的一点是：**LLM 从未亲自运行 Python 函数。**它只生成调用请求；工具分发、参数校验、真实执行和错误处理，全由你的程序负责。

放到你想做的 Hermes 类助理里也是一样：用户说“记录明早 9 点复习第 01 章”，模型请求调用 `create_todo`，你的代码写入待办，最后模型告诉用户“已记录”。

理解检查：如果 `get_weather` 发生网络超时，你觉得应该让程序直接崩溃，还是把超时错误作为工具结果发回模型？
</response>

**17:40 · Ch.03 · 超时后的确认与幂等重试**

<student>
应该先确认是否发送成功
</student>

<response>
对。正确顺序是：

1. 用原请求的 `operation_id` 查询发送状态；
2. 如果明确显示成功，直接返回成功，不再发送；
3. 如果明确显示失败，才重新发送；
4. 如果状态仍然未知，只能使用同一个 `idempotency_key` 重试，让下游服务负责去重。

关键点是：**不能把“超时”理解为“执行失败”**。超时只说明应用没有及时收到结果，邮件可能已经发送成功。

另外，“发送成功”也要定义清楚：通常只能证明邮件服务商已经接受了请求，不一定证明收件人已经收到或阅读。工具结果最好写成 `accepted`、`delivered`、`failed` 或 `unknown`，而不是含糊的 `success`。

这就是第 03 章所说的 **verify-by-read-back（写后回读验证）**；如果无法回读，就返回一个可继续查询的 `pending` 句柄。

**17:41 · Ch.03 · 幂等键的含义**

<student>
幂等键是什么
</student>

<response>
**幂等键（idempotency key）就是一次业务操作的唯一编号。**

例如，你要发送一封邮件，应用生成：

```text
idempotency_key = "send-email:trip-plan:20260618:001"
```

第一次请求：

```python
send_email(to, subject, body, idempotency_key=key)
```

邮件服务记录：这个 `key` 已经执行成功。

如果应用超时后用**相同的 key** 重试，服务发现它处理过，就返回之前的结果，不再发送第二封邮件。

```text
第一次：key=abc → 发送邮件
重试：  key=abc → 返回原结果，不重复发送
主动再发一次：key=xyz → 发送新邮件
```

它和 `tool_call_id` 的区别是：

- `tool_call_id`：帮助模型和应用匹配“哪个工具请求对应哪个结果”；
- `idempotency_key`：帮助执行系统避免同一个真实操作被重复执行。

关键规则是：**同一次操作的所有重试使用同一个 key；用户有意再次执行时使用新 key。**这就是 Ch.03 的“幂等性”。
</response>

```python
from datetime import date

from pydantic import BaseModel, Field, model_validator


class QueryTrainTicketsArgs(BaseModel):

        if self.origin.strip() == self.destination.strip():
            raise ValueError("出发站和到达站不能相同")
        if self.travel_date < date.today():
            raise ValueError("乘车日期不能早于今天")
        return self


QUERY_TRAIN_TICKETS_TOOL = {
    "name": "query_train_tickets",
    "description": (
        "查询指定出发站、到达站和乘车日期的实时火车班次及余票。"
        "当用户明确询问火车班次或余票，或者制定出差计划且需要实时火车数据时使用。"
        "不要用于购买、退票、改签、查询历史班次，也不要在只需要普通旅行建议时使用。"
        "返回匹配的车次、出发时间、到达时间和各席别余票。"
    ),
    "input_schema": QueryTrainTicketsArgs.model_json_schema(),
}
```

> 查询 2026-06-25 北京南到上海虹桥的余票。

```json
{
  "id": "call_train_001",
  "name": "query_train_tickets",
  "input": {
    "origin": "北京南",
    "destination": "上海虹桥",
    "travel_date": "2026-06-25"
  }
}
```

注意，`余票` 没有出现在 `input` 中，因为它是工具查询产生的结果。`id` 的具体字符通常由模型服务生成；应用必须原样保存，并在返回工具结果时引用它。

我建议采用**中央 dispatcher 包装**，而不是只给单个函数加装饰器：

1. **函数装饰器**：代码直观，但捕获不到处理函数运行前发生的参数校验和工具查找错误。
2. **dispatcher 统一包装（推荐）**：从参数校验到处理函数执行都在一个 `try/except` 边界内，任何普通异常都能保留原 `tool_call_id` 并返回模型。
3. **装饰器 + dispatcher 双层包装**：更适合大型系统，但对当前练习重复且难看清错误来自哪层。

修改后的 dispatcher 会捕获所有 `Exception`，但**不会把所有错误都标成可重试**：

- 参数错误：`INVALID_ARGUMENTS`，模型修正参数后可重试；
- `TimeoutError` / `ConnectionError`：`TRANSIENT_TOOL_ERROR`，可以重试；
- Schema 与处理函数不匹配：`TOOL_CONTRACT_ERROR`，不可重试，重试相同参数只会再次失败；
- 其他异常：`TOOL_EXECUTION_ERROR`，默认不可重试。

每个错误结果都包含原 `tool_call_id`、错误代码、安全的错误消息和 `retryable`。完整堆栈只写入应用日志，不发送给模型。捕获 `Exception`，但不捕获 `KeyboardInterrupt`、`SystemExit` 等 `BaseException`。

我会在现有练习中展示：

- 修改前：异常逃出 dispatcher，终止循环；
- 修改后：异常变成结构化 `tool_result`；
- 一个超时错误显示 `retryable: true`；
- 当前 Schema 漂移显示 `retryable: false`；
- 自动化测试验证两条路径。
